# 開發者實踐指南 (Developer Guide)

本指南旨在將專案的核心哲學（參見 `project_foundation.md`）轉化為具體、可執行的開發步驟。所有參與本專案的開發者都應遵循此工作流程。

---

## 核心開發五步驟流程

本專案嚴格遵循「規格驅動開發 (Spec-Driven Development)」。在動手寫任何程式碼之前，請務必完成前四個步驟。

### 1. 確立原則 (Principles)
- **目標**: 理解並認同專案的基礎規範。
- **行動**:
    - 仔細閱讀 `project_foundation.md` 以理解專案的開發哲學。
    - 熟悉 `.cursorrules` 中定義的程式碼風格與語言規範（例如：一律使用繁體中文）。

### 2. 規格定義 (Specify)
- **目標**: 清晰描述「要做什麼 (What)」和「為什麼 (Why)」。
- **行動**:
    - 建立一個新的規格文件：`specs/xxx-feature-name.md`。
    - 在文件中，專注於使用者故事、功能邊界和限制。
    - **切記**：此階段**禁止**討論技術細節（例如：要用哪個函式庫）。

### 3. 實作計畫 (Plan)
- **目標**: 詳細規劃「如何實現 (How)」。
- **行動**:
    - 建立 `implementation_plan.md` 文件。
    - 在計畫中，闡述你的技術方案：
        - 選擇的技術堆疊或函式庫。
        - 資料庫結構設計。
        - 模組與組件的架構。
    - 驗證你的計畫是技術上可行的。

### 4. 任務拆解 (Tasks)
- **目標**: 將複雜的實作計畫分解為一系列微小、可執行的任務。
- **行動**:
    - 建立 `task.md` 文件。
    - 將計畫中的每個部分，拆解成一個個的待辦事項 (Checklist)。
    - 確保每個任務都足夠小，可以獨立完成、驗證和提交。

### 5. 執行實作 (Implement)
- **目標**: 根據任務列表，高效且準確地完成程式碼撰寫。
- **行動**:
    - 逐一完成 `task.md` 中的任務。
    - 在開發過程中，隨時回頭檢視 `specs/` 文件，確保實作沒有偏離最初的規格。
    - 遵循「原子化提交」原則，每完成一個小任務就進行一次 Commit。

## 高效工作實踐 (Efficient Work Practices)

### 以程式碼為工具 (Code as a Tool)
這是本專案最重要的效率原則。當你需要處理重複、多步驟或涉及大量資料的任務時，**永遠優先考慮撰寫腳本 (Script) 來自動化處理**。

- **✅ 正確示範**:
    - **情境**: 需要分析多個 Log 檔找出錯誤根源。
    - **做法**: 撰寫一個 Python 或 Shell 腳本，自動讀取所有 Log 檔，使用正則表達式過濾關鍵錯誤，並輸出整理後的報告。

- **❌ 錯誤示範**:
    - **情境**: 同上。
    - **做法**: 手動一個個 `read_file`，然後在腦中（或對話中）進行分析和總結。這種方式不僅耗費大量 Token，也極其緩慢且容易出錯。

### 保持脈絡清潔 (Keep Context Clean)
- 在讀取檔案前，先用 `ls -R`, `find`, `grep` 等指令來探索專案結構、定位資訊。
- 只讀取你真正需要的程式碼片段，避免將整個檔案載入脈絡。
