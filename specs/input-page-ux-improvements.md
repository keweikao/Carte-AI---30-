# Input 頁面使用者體驗優化規格

## 1. 需求概述 (What & Why)

### 1.1 背景
目前 input 頁面存在以下使用者體驗問題：
1. 頁面標題「客製化你的餐點」不夠吸引人，未能傳達「饗宴」的高級感
2. 預算類型選擇器（每人/總預算）視覺上不明顯，使用者難以察覺可以切換
3. 預算金額預設為 0，使用者不知道可以手動輸入
4. 飲食偏好的命名和選項設計不當，容易造成誤解（例如：5人中只有1人吃素，選了「素食」可能被理解為全員素食）
5. 缺少返回上一頁的導航功能

### 1.2 目標
優化 input 頁面的使用者體驗，使其：
- 更符合「饗宴」的情境定位
- 提供清晰的互動提示
- 避免選項造成的語意混淆
- 改善導航流程

---

## 2. 功能需求 (Requirements)

### 2.1 頁面標題優化

**現況**：客製化你的餐點

**需求**：更換為更有吸引力、符合饗宴情境的標題

**標題候選方案**：
1. 開始設計你的完美饗宴
2. 打造專屬你的美食時光
3. 量身訂製您的餐桌盛宴
4. 設計一場完美的餐聚體驗
5. 開啟你的美食探索之旅

**決策依據**：
- 傳達「高級感」與「客製化」
- 符合多人聚餐的情境
- 避免過於商業化或過於文青

---

### 2.2 預算類型選擇器視覺優化

**現況**：
```
[每人(客單)] [總預算]  // 視覺上不明顯是可點擊的
```

**需求**：
- 加強可點擊的視覺提示
- 讓使用者明確知道這是「選擇器」而非「標籤」
- 保持與整體設計風格一致

**設計方向**：
- 增加 hover 效果
- 未選中的選項要有明顯的「可點擊」狀態
- 選中的選項要有更強的視覺對比
- 可考慮加入圖示（如：人形圖示 / 總和圖示）

---

### 2.3 預算金額輸入優化

**現況**：
- 預設值：0 元
- 無輸入提示

**需求**：
- 移除預設值 0，改為 placeholder 提示
- 提供清晰的輸入引導
- 保留原有的 slider 功能

**設計方案**：
- Placeholder 文字：「請輸入預算金額」或「例如：500」
- 輸入框增加聚焦狀態的視覺提示
- 保持與 slider 的雙向同步

---

### 2.4 飲食偏好重新設計

**現況問題分析**：
- 標題：「飲食偏好」- 不符合饗宴情境
- 選項：「素食」、「海鮮」等 - 看起來像是非題（全選或不選）
- 語意混淆：5人中1人吃素，選「素食」會被誤解為全員素食

**需求**：
- 重新命名標題，更符合「饗宴」情境
- 重新設計選項，避免「全有或全無」的誤解
- 選項應該表達「偏好程度」或「組成比例」

**設計方案**：

#### 標題候選：
1. 用餐風格偏好
2. 菜色搭配喜好
3. 餐桌上的堅持
4. 特別的飲食需求

#### 選項重新設計：

**原選項 → 新選項**：
- 素食 → 「鍋邊素」或「需要素食選項」
- 海鮮 → 「多點海鮮」或「愛吃海鮮」
- 肉類 → 「愛吃肉」或「多點肉類」
- 健康 → 「多蔬菜少油」或「偏好清淡」
- 辣 → 「能吃辣」或「偏好辣味」

**新選項集合建議**：
1. 愛吃肉（多點肉類料理）
2. 多點海鮮（偏好海鮮類）
3. 需要素食選項（有人吃素）
4. 多蔬菜（增加蔬菜比例）
5. 偏好清淡（少油少鹽）
6. 能吃辣（可以有辣味菜）
7. 避免過敏原（海鮮/堅果等）

**設計原則**：
- 使用「多點」、「偏好」、「需要」等詞彙，表達「比例」而非「全有」
- 括號內補充說明，避免歧義
- 多選機制，允許同時選擇多個偏好

---

### 2.5 返回上一頁功能

**現況**：無返回上一頁按鈕

**需求**：
- 新增「返回上一頁」或「上一步」按鈕
- 位置：頁面左上角或底部導航區
- 功能：保留目前填寫的表單資料（如果可行）

**設計方案**：
- 左上角：加入 `<ArrowLeft>` 圖示 + 「返回」文字
- 點擊行為：`router.back()` 或導航至首頁
- 考慮是否需要確認對話框（避免使用者誤觸後遺失填寫內容）

---

## 3. 非功能需求

### 3.1 可用性
- 所有互動元素應符合無障礙標準（ARIA 標籤）
- 響應式設計：確保在手機、平板、桌面都有良好體驗
- 視覺回饋：hover、focus、active 狀態都要明確

### 3.2 一致性
- 保持與現有設計系統的一致性（顏色、字體、間距）
- 保持與 recommendation 頁面的設計語言一致

### 3.3 效能
- 不影響現有頁面載入速度
- 表單驗證即時反饋

---

## 4. 邊界與限制

### 4.1 不包含的項目
- 不修改表單的資料結構（仍使用現有的 API 格式）
- 不修改後端 API

### 4.2 技術限制
- 保持 Next.js 14 App Router 架構
- 保持現有的 shadcn/ui 元件庫
